# [Bug] Missing numerical range validation in POST /user/food-intake

## Metadata
- **ID:** #7
- **Reporter:** Vyskrebstsov O.
- **Severity:** Major
- **Priority:** High
- **Type:** Functional
- **Environment:** Windows 10 Pro, Google Chrome 146.0.7680.80, API v1.0.0, OAS 3.1

---

## Summary
The `POST /user/food-intake` endpoint accepts physically impossible numerical values (negative and extreme) for nutritional fields.

## Description
The API accepts and processes invalid values (e.g., numeric strings for names and extremely large integers for nutritional values) without returning an error. This results in corrupted data being stored and returned in the response.

## Pre-conditions
1. Open the API documentation: <https://healthy-hub-qa.b.goit.study/api-docs/#/>
2. Ensure the user is authorized with a valid Bearer token.

## Steps to Reproduce
1. Locate the [POST /user/food-intake](https://healthy-hub-qa.b.goit.study/api-docs/#/User/post_user_food_intake) endpoint.
2. Click the **"Try it out"** button.
3. In the request body, insert the following JSON with invalid values:
   ```json
   { "breakfast": [ { "name": "Apple", "calories": -500, "carbohydrates": 555555555555, "protein": -300, "fat": 0.2 } ] }
   ```
5. Click the "Execute" button.

## Actual Result
- **Status Code:** 200 OK
- **Response Body:** The server saves negative and extreme values, leading to corrupted "daily total" statistics (e.g., total.calories.used becomes negative).

## Expected Result
- **Status Code:** 400 Bad Request
- **Behavior:** The system should validate that nutritional values fall within a realistic and non-negative range (e.g., min: 0, max: 10000).
-The server should return a clear validation error message.

Similar data validation issues are observed in the [POST /api/user/food-intake-v2](https://healthy-hub-qa.b.goit.study/api-docs/#/User/post_user_food_intake_v2) endpoint. Processing such large or negative numbers may lead to unpredictable behavior during subsequent data processing or problems with user interface rendering.
