# [Bug] POST /api/user/food-intake behaves like GET when sending an empty JSON object

## Metadata
- **ID:** #12
- **Reporter:** Vyskrebstsov O.
- **Severity:** Major
- **Priority:** Medium
- **Type:** Functional
- **Environment:** Windows 10 Pro, Google Chrome 146.0.7680.80, API v1.0.0, OAS 3.1

---

## Summary
The `POST /api/user/food-intake` endpoint incorrectly returns current user statistics (behaving like a `GET` request) instead of validating the request when an empty JSON object `{}` is provided in the request body.

## Description
When a `POST` request is sent with an empty body `{}`, the server returns a `200 OK` status code and displays the user's daily statistics. No new record is created, and no validation errors are triggered for missing mandatory fields. This behavior violates REST standards for `POST` methods.

## Pre-conditions
1. Open the API documentation: <https://healthy-hub-qa.b.goit.study/api-docs/#/>
2. Ensure the user is authorized with a valid Bearer token.

## Steps to Reproduce
1. Locate the `POST /api/user/food-intake` endpoint.
2. Click the **"Try it out"** button.
3. In the **"Request body"**, enter an empty JSON object: `{}`.
4. Click the **"Execute"** button.

## Actual Result
- **Status Code:** 200 OK.
- **Response Body:** Displays current daily statistics for the user.
- **Behavior:** The endpoint acts as a retrieval method (`GET`) rather than a creation method.

## Expected Result
- **Status Code:** 400 Bad Request.
- **Behavior:** The server should validate the request and return an error indicating that mandatory fields (e.g., `"name"`, `"calories"`) are missing.

## Additional Info
- The same issue is observed in the `POST /user/food-intake-v2` endpoint.
- This behavior suggests that the backend handler might be defaulting to a "fetch" logic when it fails to parse or find data to save, instead of throwing a validation exception.
