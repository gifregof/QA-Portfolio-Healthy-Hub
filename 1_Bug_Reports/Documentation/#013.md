# [Minor] Response Schema Mismatch in Swagger for GET /user/food-intake (String instead of Object)

## Metadata
- **ID:** #13
- **Reporter:** Vyskrebstsov O.
- **Severity:** Minor
- **Priority:** Medium
- **Type:** Functional / Documentation
- **Environment:** Windows 10 Pro, Google Chrome 146.0.7680.80, API v1.0.0, OAS 3.1

---

## Summary
The `GET /user/food-intake` endpoint's documentation incorrectly displays the response "Example Value" as a `string`, whereas the actual API returns a structured JSON object.

## Description
In the Swagger/OAS 3.1 documentation, the response for this endpoint is poorly defined. Because the underlying schema (`GetFoodIntakeResponse`) uses the `any` type for its main fields, Swagger UI defaults to displaying the response example as a simple string. This prevents users from seeing the actual structure of the returned data.
Note: Verification was performed via Postman because the current Swagger endpoint URL is incorrect.

## Pre-conditions
1. Open the API documentation: <https://healthy-hub-qa.b.goit.study/api-docs/#/>
2. Ensure the user is authorized with a valid Bearer token.

## Steps to Reproduce
1. Locate the [GET /user/food-intake](https://healthy-hub-qa.b.goit.study/api-docs/#/User/get1_user_food_intake) endpoint.
2. Observe the **"Responses"** section and the **"Example Value"** provided.
3. Compare the documentation example with a real successful request execution.

## Actual Result
- The **"Example Value"** is displayed as a simple `string`.
- In the **"Schemas"** section, the `GetFoodIntakeResponse` fields (e.g., `breakfastTotal`, `breakfastDishes`) are defined with the type `any`.

## Expected Result
- The **"Example Value"** should accurately reflect the JSON structure returned by the server (including fields like `breakfastTotal`, `breakfastDishes`, etc.).
- The schema should explicitly define the data types for each field instead of using `any`.
