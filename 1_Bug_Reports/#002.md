	# [Critical] POST /auth/restore returns 401 Unauthorized instead of 200 OK

## Metadata
- **ID:** #4
- **Reporter:** Vyskrebstsov O.
- **Severity:** Critical
- **Priority:** High
- **Type:** Functional
- **Environment:** Windows 10 Pro, Google Chrome 146.0.7680.80, API v1.0.0, OAS 3.1

---

## Summary
The `POST /auth/restore` endpoint incorrectly requires authorization, returning a `401 Unauthorized` status code. Additionally, the password reset logic appears to lock users out of their accounts.

## Description
The password restoration endpoint should be public and not require an authorization token. Currently, it returns 401. Furthermore, after attempting a restoration, the user loses access to their account, and subsequent login attempts fail with an "Incorrect email or password" error.

## Pre-conditions
1. Open Swagger UI: `https://healthy-hub-qa.b.goit.study/api-docs/#/`
2. Ensure the user is authorized with a valid token (though this endpoint should not require it).

## Steps to Reproduce
1. Locate the `POST /auth/restore` request.
2. Click the **"Try it out"** button.
3. Enter a valid registered email address in the "email" field.
4. Click the **"Execute"** button.

## Actual Result
* **Status Code:** 401 Unauthorized.
* **Message:** "Unauthorized".
* **Side Effect:** The user loses access to the account. Attempting to log in with the old credentials results in a `401` error with the message "Incorrect email or password".

## Expected Result
* **Status Code:** 200 OK.
* **Message:** "New password sent to email".
* **Behavior:** The endpoint should be accessible without an authorization header. The old password should only be invalidated once the new one is successfully applied or sent.

## Additional Info
The endpoint likely has incorrect middleware settings regarding authorization. The immediate lockout suggests the system updates the password in the database before ensuring the user can actually receive or use the new credentials.
